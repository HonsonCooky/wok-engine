# Sharp Edges

Traps hit while building the editor and the substrate it sits on, written so the next orchestrator + CC combo does not rediscover them. This is not design (see `high-level-design.md`, `project-canon.md`, `editor-design.md`); it is the list of things that bit, the symptom each produced, and the rule that avoids it. The editor crate (`wok`) has been cleared to a shell and is being rebuilt; the substrate (`wok-platform`, `wok-render`) and these lessons survive the reset.

## 1. The frame loop and resize (wok-platform + wok-render)

The editor's render path sits on `wok_platform::gfx::begin_frame`. This is where the worst bug lived; rebuild the render path against these rules.

- **One size per frame, taken from the acquired surface texture.** Size the depth buffer and the render viewport from `Frame::size()` (the size of the texture `begin_frame` just acquired), never from `window.inner_size()` or any separately tracked size. Two size sources update on different clocks: `window.inner_size()` is read fresh each frame, while the surface is reconfigured only on the lagging `WindowEvent::Resized`. During a live drag the window size races ahead, so a frame binds a depth buffer of one size against a colour target of another and the forward pass panics with wgpu's "Attachments have differing sizes". The fix lives in `wok-platform/src/gfx/mod.rs` (`begin_frame` + `Frame::size`) and `wok-render` (`Renderer::resize` is idempotent, called every frame from `Frame::size`); commit `c35ac01`.
- **`begin_frame` is the single home for frame acquisition.** It acquires the texture, handles a `Lost`/`Outdated` surface (reconfigure to the current config and re-acquire once, else skip the frame), and never holds a `SurfaceTexture` across the reconfigure. Do not scatter `get_current_texture` calls; go through `begin_frame`.
- **Reproduction trap: scripted resizes do not reproduce the divergence.** On Windows `window.request_inner_size` is asynchronous (returns `None`) and keeps `inner_size` and `surface_config` in lockstep, so oscillating the size in code never trips the bug. Only a real hand-drag (the OS modal resize loop) does. "It does not reproduce in normal use" did not mean "no bug"; it meant the synthetic path hid it. Reproduce on the real input path, or force the exact failing condition directly (set the depth to a stale size for one frame, then render).
- **The default backend here is Vulkan, not DX12.** `Backends::PRIMARY` selected `Vulkan` on the dev machine (Intel Arc integrated GPU). Print the chosen backend at runtime when diagnosing GPU issues; do not assume DX12.
- **Vulkan turns any frame-time panic into a confusing second panic.** wgpu-hal Vulkan asserts on surface teardown that no acquired `SurfaceTexture` is still alive ("Trying to destroy a SurfaceSemaphores that is still in use by a SurfaceTexture"). A panic in the render pass unwinds with the acquired texture still live, so it chains into this teardown panic and buries the real cause. It is a symptom, not a separate bug. The rule that prevents it: every acquired texture is presented (`Frame::finish`) on every path, and a skip never strands one. Check the FIRST panic, not the teardown panic.

## 2. egui chrome patterns

The chrome is egui (0.31), an application-only dependency. These are the layout/paint gotchas the chrome rebuild will hit.

- **Panels carry a frame inner margin.** `TopBottomPanel`/`SidePanel` default to `Frame::side_top_panel(style)`, whose `inner_margin` (symmetric 8, 2) offsets the content from the panel edge. To make content sit flush, pass a frame with `inner_margin(Margin::ZERO)` but keep the themed fill: `Frame::side_top_panel(&ctx.style()).inner_margin(Margin::ZERO)`. `Frame::NONE` drops the fill too, which exposes the renderer's clear colour through the strip.
- **Full-height square cells, padding for separation.** The simple toolbar/tab model: each cell fills the bar height, icon buttons are square (`min_size = (available_height, available_height)`), and the gap between the glyph and the square's edges is the separation. Do not add explicit gaps plus custom hover paint; that was the overcomplicated path. Center content in a full-height cell with `ui.set_min_height(available_height)` + `horizontal_centered`.
- **Hover and active fills come free from `Button`.** The theme sets `widgets.hovered.weak_bg_fill`, so any `egui::Button` lights up on hover with no custom code; use a button for anything button-shaped. When a fill must depend on a hover test that is only known after layout (e.g. a tab cell that is not a button), reserve a paint slot before the content and set it after: `let bg = ui.painter().add(Shape::Noop); ...; ui.painter().set(bg, Shape::rect_filled(rect, 0.0, fill))`. Paint behind, not on top.
- **`exact_height` vs content height.** A top panel clips its fill to `exact_height` but reserves the larger content-driven height for the central panel below. If the row content overflows `exact_height`, an unpainted strip shows the renderer's clear colour where the chrome should be. Size the bar so the content fits with no overflow.
- **Region order: the nav panel is added before the view column.** The status bar belongs to the view column only, never under the nav (README + editor-design). egui allocates panel space in the order panels are added, so add the full-height nav `SidePanel` FIRST; the view-column stack (the tab-bar `TopBottomPanel`, the status-bar `TopBottomPanel`, then the `CentralPanel`) fills what remains. c35ac01 added `status_bar` before the nav `SidePanel`, which stretched the status bar full-width under the nav. When lifting chrome from c35ac01 for later slices, reorder: nav first, then the view column (fixed in 9a7aafa's `view::chrome`).
- **Nerd Font icons fill the em; size them below text px.** A Nerd Font glyph paints across the whole em box, so an icon at font-size N is a solid ~Npx mark - visually far larger than Npx text, whose caps are ~0.7 em. Matching icons to the text size makes them read comically large. The chrome normalizes every painted glyph to a uniform ink height of 12px (`icons::SIZE`), measured and rescaled in `icons::paint` then centred on the ink rather than the em, because MDI glyphs fill the em by different amounts so a single uniform font size (the prior settle at da59c5a) still rendered them at visibly different sizes; size future icons against the visual mark, not the text px (the tab close glyph is a sized label, outside `icons::paint`).
- **Palette field visibility.** Colours the view paints directly must be `pub` on the palette; colours that only feed egui's own widget visuals can stay private. Reuse the same `hover` colour egui gives buttons for any hand-painted hover fill, so the affordances match.
- **Viewport input gating.** The 3D viewport is egui's background layer under a `CentralPanel`, so `is_pointer_over_area` / `wants_pointer_input` are always true over it and cannot tell the viewport from a panel. Gate viewport camera input on the editor rect itself, plus excluding any foreground layer over it (an open menu) and `is_using_pointer` (a panel-resize drag straying onto the viewport).
- **Cursor-lock engage, and the `Confined` warp (camera drag).** Two traps the camera-drag cursor lock hit (`9e36a85`, fixed in `8ebb202`). (1) Engage the lock on "pointer over the well rect, no foreground layer" but NOT on `is_using_pointer` / `pointer_free`: the well runs a background click-sense (the empty-well deselect), so egui sets `is_using_pointer` on the press frame and only clears it a few pixels later, so gating the press-edge engage on it leaves a dead spot or misses the press entirely. A press lands over the well only for a real viewport drag (a panel resize presses the panel edge), so dropping `is_using_pointer` for the engage is safe; also OR the lock state into the camera gate so the camera drives from the press frame too. (2) Windows grants `CursorGrabMode::Confined`, not `Locked` - `Confined` does not freeze the cursor, so re-warp it to the anchor and re-assert `set_cursor_visible(false)` every frame while locked. No `CursorMoved` discard is needed: the camera reads raw `DeviceEvent::MouseMotion`, which a synthetic warp never produces.
- **Pins.** `egui` / `egui-wgpu` / `egui-winit` are `0.31`, matched to `wgpu 24` and `winit 0.30` in the workspace root `Cargo.toml`. egui is a `wok`-only dependency; no engine crate may depend on it.

## 3. Snapshot tests and verification

- **The snapshot harness is the screenshot tool.** `egui_kittest` (wgpu + snapshot features) renders the chrome through the same view functions the app calls, to a PNG. For any visual change, render the snapshot and read the PNG back to verify - do not reason about pixels blind. Refresh intended changes with `UPDATE_SNAPSHOTS=1 cargo test -p wok` and commit the new PNG. Both the live app and the test must render through one composition root so the PNG tracks the real chrome.
- **Shared style/font setup, or the snapshot tofus.** The harness builds its own egui context through the shared view/theme code, not the live app's startup path, so any font or style the chrome needs must be installed where both reach it. The icon Nerd Font is registered in `theme::apply` for exactly this reason; registering it only on the app's launch path would leave the snapshot rendering missing-glyph tofu (e2fe343).
- **Serialize GPU tests.** Hold a single process-wide mutex (`GPU_TEST_LOCK`) across each wgpu snapshot test. Creating/tearing several headless wgpu devices concurrently crashes on Windows - the same Vulkan teardown race as above. Recover from a poisoned lock so one failed assert does not cascade.
- **accesskit feature unification.** kittest queries the accessibility tree, which turns on egui's `accesskit` feature; Cargo's feature unification then needs `egui-winit`'s matching `accesskit` path compiled too. Enable `egui-winit` with `features = ["accesskit"]` in `[dev-dependencies]` (resolver 3 keeps it out of a normal `cargo build`). This is preserved in `wok/Cargo.toml`; do not remove it or the snapshot build breaks with an obscure error.
- **Construct states the live app cannot reach.** Some affordances (e.g. an inactive-tab hover) need a state the single-tab app never shows. Build the model directly in a test (`Shell::open_tab` twice) and drive `harness.get_by_label(..).hover()` to exercise it. egui opens submenus on hover, not on an accesskit click, so descend menus with `.hover()`.
- **Runtime verification of windowed behaviour.** For things only a running window exercises (resize, teardown), drive the live binary with a temporary in-`frame` harness (oscillate the size, force the failing condition once, request a clean close after N frames) and grep the captured log for panics. Remove the harness before committing. The resize race needs a real drag, so a forced-condition one-shot is the deterministic proxy.

## 4. Process and brief-writing

- **Reproduce on the real input path before fixing.** Distinguish synthetic input (scripted resize, generated events) from real OS input; timing bugs hide on the synthetic path. A brief that says "reproduce first, report the backend and the exact error" was correct and saved a blind patch.
- **Annotated screenshots: the ink is markup, not UI colour.** When the user draws red/blue on a screenshot, red means "remove this" and blue means "grow/fill this" - they are not asking for a red or blue UI. Confirm the rendering-versus-annotation reading early instead of theorising about colours that are not in the code.
- **Do not over-engineer.** When the user says "it's simple", lean on the framework's built-in affordances (square buttons with natural hover, selectable labels) over custom painting and bespoke spacing. The simplest version that uses the framework's grain is usually what they want.
- **Hand-formatted, ~400-line files.** Rust here is hand-formatted to 120 columns; do not run `cargo fmt` (the committed baseline diverges from rustfmt defaults and it churns untouched files). Keep files near the 400-line target. Documents in `designs/` are one line per paragraph or list item (not hard-wrapped); do not rewrap them.
- **The editor gets reset; capture lessons where they survive.** The editor has been rebuilt several times. Conversational lessons evaporate; put load-bearing decisions in canon and implementation traps here, so a reset does not also reset the hard-won knowledge.

## 5. Platform input (wok-platform)

The input collector (`wok-platform/src/input/mod.rs`) snapshots keys and buttons each frame. The trap here stuck a key "down" forever; fixed in `bc241ee` with regression tests, and the rules below keep it fixed.

- **Track printable keys case-folded, not by the raw `logical_key`.** winit's `logical_key` reflects the LIVE modifier state, so a letter pressed without Shift arrives as `Character("a")` but its release while Shift (or a held boost) is down arrives as `Character("A")` - a different key. `keys_held.remove("A")` then misses the `"a"` still in the set and the key stays "held" forever (symptom: the editor's camera kept strafing with nothing pressed, hit through the W1 Shift-boost). The fix lowercases printable characters at the tracking edge (`Key::Character(s.to_ascii_lowercase())`) so a press and its release always land on the same key. Every accessor already matched case-insensitively, so nothing else changed and taste's Level 2 replay still reproduces bitwise.
- **Clear the held sets on focus loss.** A key or button still down when the window loses focus (alt-tab to the terminal) never sees its release - that event lands in the other window - so it sticks. `WindowEvent::Focused(false)` clears `keys_held` and `mouse_buttons_held` back to neutral; the per-frame edge sets (pressed / released / repeating) clear themselves at snapshot, so only the held sets need it.
- **Held SYMBOL keys across a Shift change are still a latent desync.** The lowercase fold covers letters only; a held `1` that releases as `!` (Shift down) still mismatches and sticks. Nothing today holds a symbol key (movement and hotkeys are letters; the stamp row 1-5 is a press edge, not held), so it is deferred under earn-its-place. The real fix when a held symbol-key action appears is to track by `physical_key` (the modifier-independent `KeyCode` position) rather than the logical key - a bigger change, not worth it until then.
